# LLM Config Center 开发文档

## 1. 项目背景

当前多个后端服务都会直接配置大模型相关参数，例如：

```text
model
api_key
base_url
provider
temperature
max_tokens
timeout
stream
```

当大模型服务升级、更换供应商、更换 API Key、切换模型或调整默认参数时，需要逐个修改各个后端服务配置，维护成本高，且容易出错。

本项目要实现一个内部使用的 **大模型配置中心**，统一管理大模型供应商、模型、上游 API Key、模型别名、默认参数、应用权限和访问密钥。

业务后端服务不再本地写死真实模型配置，而是通过配置中心 Runtime API 获取有权限的模型配置，然后在本地初始化大模型客户端。

------

## 2. 项目名称

```text
llm-config-center
```

中文名称：

```text
大模型配置中心
```

------

## 3. 一句话定位

LLM Config Center 是一个面向内部后端服务的大模型配置中心，提供 Web 管理后台统一维护供应商、模型、上游 API Key、模型别名和默认参数；业务服务通过 Access Key 调用 Runtime API 获取有权限的模型配置，并在本地初始化大模型客户端。

------

## 4. 核心目标

### 4.1 解决的问题

1. 多个后端服务重复配置大模型信息。
2. 更换 API Key 时需要逐个服务修改。
3. 更换 base_url 或模型名时需要逐个服务修改。
4. 不同环境 dev、test、prod 配置不统一。
5. 缺少统一的模型配置版本管理。
6. 缺少统一的应用访问权限控制。
7. 缺少配置变更审计和回滚基础。

### 4.2 第一版目标

第一版做一个轻量但完整的配置中心，包含：

1. Web 管理后台。
2. 账号密码登录。
3. Provider 管理。
4. 上游模型 API Key 管理。
5. Model 管理。
6. Alias 模型别名管理。
7. App 应用管理。
8. App Access Key 管理。
9. App 对 Alias 的读取权限管理。
10. Runtime API 配置下发。
11. Access Key 鉴权。
12. 上游 API Key 加密存储。
13. 配置版本机制。
14. 操作审计日志。
15. Python SDK。

------

## 5. 非目标

第一版暂不做：

1. 不做代理转发模型请求。
2. 不统计每次模型调用的 token。
3. 不统一限流真实模型调用。
4. 不做模型调用 fallback。
5. 不做复杂 RBAC 权限系统。
6. 不做多租户隔离。
7. 不做 K8s 部署。
8. 不做企业 SSO。
9. 不做复杂审批流。

------

## 6. 系统总体架构

```text
                  ┌────────────────────────┐
                  │      Web 管理后台       │
                  │ 登录 / 配置 / 创建 Key  │
                  └───────────┬────────────┘
                              │
                              ▼
                    ┌────────────────────┐
                    │  LLM Config Center  │
                    │   FastAPI Backend   │
                    └─────────┬──────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   PostgreSQL              Redis              Audit Log
 存用户/配置/权限       缓存配置/版本         操作审计记录

                              ▲
                              │
                              │ Runtime API
                              │ Access Key 鉴权
                              │ 配置下发
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
  后端服务 A              后端服务 B              后端服务 C
  拉取配置                拉取配置                拉取配置
  初始化模型              初始化模型              初始化模型
```

业务服务调用链路：

```text
业务服务启动
  ↓
使用 Access Key 请求 Runtime API
  ↓
配置中心校验 Access Key
  ↓
校验 App 是否有权限读取目标 alias
  ↓
返回 provider / model / base_url / api_key / params
  ↓
业务服务本地初始化 OpenAI-compatible Client
  ↓
业务服务直连真实大模型供应商
```

## 7. 核心概念

### 7.1 Provider

大模型供应商。

例如：

```text
volcengine
dashscope
deepseek
zhipu
siliconflow
openai
local_vllm
```

每个 Provider 主要包含：

```text
code
name
protocol
base_url
status
description
```

------

### 7.2 Provider API Key

真实上游大模型服务的 API Key。

例如：

```text
火山 API Key
阿里百炼 API Key
DeepSeek API Key
OpenAI API Key
本地 vLLM Token
```

要求：

1. 数据库中必须加密存储。
2. 后台列表只展示脱敏值。
3. Runtime API 只有权限校验通过后才可以解密并返回。
4. 审计日志中禁止记录明文 key。

------

### 7.3 Model

供应商下面的具体模型。

例如：

```text
doubao-seed-1.6
qwen-plus
deepseek-chat
gpt-4.1
text-embedding-v3
```

模型包含：

```text
model_name
model_type
support_stream
support_vision
context_window
max_output_tokens
status
```

------

### 7.4 Alias

模型别名，是业务服务真正使用的模型名称。

业务服务不应该直接依赖真实模型名，而应该依赖 Alias。

例如：

```text
chat-default
chat-fast
chat-cheap
chat-high-quality
vision-default
embedding-default
rerank-default
image-generate-default
```

Alias 映射到：

```text
env
model
provider_api_key
default_params
version
status
```

示例：

```text
prod / chat-default
  provider: volcengine
  model: doubao-seed-1.6
  api_key: 火山生产 Key 1
  base_url: https://ark.cn-beijing.volces.com/api/v3
```

以后要切到阿里，只改 Alias 绑定关系：

```text
prod / chat-default
  provider: dashscope
  model: qwen-plus
  api_key: 阿里生产 Key 1
  base_url: https://dashscope.aliyuncs.com/compatible-mode/v1
```

业务服务无需修改代码。

------

### 7.5 App

调用配置中心的内部后端服务。

例如：

```text
requirement-api
design-api
crawler-api
admin-agent
message-reply-api
```

每个 App 可以拥有多个 Access Key。

------

### 7.6 Access Key

App 调用 Runtime API 时使用的访问凭证。

Access Key 不是上游模型 API Key。

Access Key 用于访问配置中心，例如：

```text
lcg_ak_9f8a7b6c.xxxxxxxxxxxxxxxxxxxxx
```

要求：

1. 创建后只展示一次完整明文。
2. 数据库只存 hash。
3. 后台列表只展示 prefix 和脱敏信息。
4. 可以禁用。
5. 可以设置过期时间。
6. 可以记录最近使用时间。
7. 可以按 App 控制权限。

------

### 7.7 Permission

控制某个 App 可以读取哪些 Alias。

例如：

```text
requirement-api 可以读取：
- prod / chat-default
- prod / embedding-default

design-api 可以读取：
- prod / chat-default
- prod / vision-default
- prod / image-generate-default
```

------

### 7.8 Config Version

配置版本。

每个环境有一个全局版本号。

只要会影响 Runtime 配置返回结果的数据发生变化，就需要让对应环境的版本号递增。

包括：

```text
provider 修改
provider_api_key 修改
model 修改
alias 修改
permission 修改
```

客户端 SDK 可通过版本号判断是否需要刷新配置。

## 10. 后端数据库设计

数据库使用 PostgreSQL。

### 10.1 用户表 sys_user

```sql
CREATE TABLE sys_user (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(100) UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    display_name VARCHAR(100),
    role VARCHAR(50) DEFAULT 'operator',
    status VARCHAR(20) DEFAULT 'enabled',
    last_login_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

字段说明：

| 字段          | 说明                                                |
| ------------- | --------------------------------------------------- |
| username      | 登录用户名                                          |
| password_hash | 密码哈希                                            |
| role          | super_admin / admin / app_owner / viewer / operator |
| status        | enabled / disabled                                  |

------

### 10.2 Provider 表 llm_provider

```sql
CREATE TABLE llm_provider (
    id BIGSERIAL PRIMARY KEY,
    code VARCHAR(100) UNIQUE NOT NULL,
    name VARCHAR(100) NOT NULL,
    protocol VARCHAR(50) DEFAULT 'openai_compatible',
    base_url TEXT NOT NULL,
    status VARCHAR(20) DEFAULT 'enabled',
    description TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

字段说明：

| 字段     | 说明                                   |
| -------- | -------------------------------------- |
| code     | 供应商编码，例如 volcengine            |
| name     | 供应商名称，例如 火山引擎              |
| protocol | 协议类型，第一版默认 openai_compatible |
| base_url | 供应商基础 URL                         |
| status   | enabled / disabled                     |

字段说明：

| 字段              | 说明                          |

## 11. 鉴权设计

### 11.1 后台登录鉴权

后台登录使用账号密码。

密码存储：

```text
argon2 或 bcrypt hash
```

登录成功后返回后台访问 token。

推荐：

```text
HttpOnly Cookie + JWT
```

第一版也可以使用：

```text
Authorization: Bearer admin_jwt
```

后台接口统一依赖当前登录用户。

### 11.2 后台角色

第一版支持以下角色：

| 角色        | 权限                           |
| ----------- | ------------------------------ |
| super_admin | 所有权限，包括用户管理         |
| admin       | 管理模型配置、App、Key、权限   |
| app_owner   | 管理自己负责 App 的 Access Key |
| viewer      | 只读                           |
| operator    | 普通配置操作员                 |

第一版可以先简化为：

```text
super_admin
admin
viewer
```

### 11.3 Runtime API 鉴权

业务服务调用 Runtime API 时必须携带 Access Key：

```http
Authorization: Bearer lcg_ak_9f8a7b6c.xxxxxxxxx
```

验证流程：

```text
1. 从 Authorization header 解析 token
2. 校验 token 格式是否为 lcg_ak_{prefix}.{secret}
3. 根据 prefix 查询 llm_app_access_key
4. 判断 access key 是否存在
5. 判断 access key 是否 enabled
6. 判断 access key 是否过期
7. 对 secret 做 hash
8. 和数据库 key_hash 比对
9. 查询所属 app
10. 判断 app 是否 enabled
11. 更新 last_used_at
12. 返回当前 app 上下文
```

## 12. 加密设计

### 12.1 上游 API Key 加密

上游 API Key 需要可逆加密，因为 Runtime API 要返回给有权限的后端服务。

使用：

```text
AES-GCM 或 Fernet
```

第一版推荐使用 Python cryptography 的 Fernet。

环境变量：

```bash
LLM_CONFIG_MASTER_KEY=base64_fernet_key
```

要求：

1. 加密主密钥不入库。
2. 加密主密钥不提交代码。
3. .env.example 只放示例，不放真实值。
4. 后台展示只展示 key_mask。
5. 审计日志不记录明文。

### 12.2 Access Key 哈希

Access Key 不需要可逆，因此只存 hash。

建议：

```text
sha256(secret + salt)
```

或者直接使用：

```text
HMAC-SHA256
```

环境变量：

```bash
ACCESS_KEY_HASH_SALT=xxxxx
```

创建 Access Key 时：

```text
prefix = 8 位随机字符串
secret = 高强度随机字符串
full_key = lcg_ak_{prefix}.{secret}
key_hash = hmac_sha256(secret, ACCESS_KEY_HASH_SALT)
```

创建后只返回一次 full_key。

## 14. 后台 API

### 14.1 登录

```http
POST /api/v1/admin/auth/login
```

请求：

```json
{
  "username": "admin",
  "password": "123456"
}
```

响应：

```json
{
  "access_token": "admin_jwt",
  "token_type": "bearer",
  "user": {
    "id": 1,
    "username": "admin",
    "display_name": "管理员",
    "role": "super_admin"
  }
}
```

------

### 14.2 获取当前用户

## 17. 服务端核心逻辑

### 17.1 创建上游 API Key

流程：

```text
1. 校验当前用户已登录
2. 校验用户有权限
3. 接收明文 api_key
4. 生成 key_mask
5. 使用 master key 加密 api_key
6. 保存 encrypted_api_key 和 key_mask
7. 写入审计日志
8. 返回脱敏结果
```

------

### 17.2 创建 Access Key

流程：

```text
1. 校验当前用户已登录
2. 校验用户有权限管理该 App
3. 生成 prefix
4. 生成 secret
5. full_key = lcg_ak_{prefix}.{secret}
6. key_hash = HMAC-SHA256(secret, salt)
7. 保存 key_prefix 和 key_hash
8. 写入审计日志
9. 返回 full_key
```

------

### 17.3 Runtime 获取配置

流程：

```text
1. 解析 Authorization Bearer Token
2. 校验 Access Key
3. 获取 App 上下文
4. 查询 alias + env
5. 校验 App 权限
6. 校验 alias enabled
7. 查询 model
8. 校验 model enabled
9. 查询 provider
10. 校验 provider enabled
11. 查询 provider_api_key
12. 校验 provider_api_key enabled and not expired
13. 解密 provider_api_key
14. 组装响应
15. 更新 access_key.last_used_at
16. 返回配置
```

------

### 17.4 配置版本更新

任何会影响 Runtime 配置的操作，都调用：

```text
bump_config_version(env)
```

影响操作包括：

```text
创建 / 修改 / 禁用 / 启用 alias
修改 provider base_url
禁用 / 启用 provider
修改 model
禁用 / 启用 model
修改 provider_api_key
禁用 / 启用 provider_api_key
新增 / 删除 app_permission
```

## 24. 安全要求

### 24.1 必须实现

1. 密码 hash 存储。
2. Access Key hash 存储。
3. 上游 API Key 加密存储。
4. 后台 API 需要登录。
5. Runtime API 需要 Access Key。
6. App 只能读取自己有权限的 Alias。
7. 审计日志不能记录明文密钥。
8. key 创建后只展示一次。
9. disabled 的 provider / model / api_key / alias 不允许被 Runtime API 返回。
10. 过期的 Access Key 不允许调用 Runtime API。

### 24.2 建议实现

1. 登录失败次数限制。
2. Runtime API IP 白名单。
3. HTTPS。
4. Nginx 内网访问限制。
5. 管理员操作二次确认。

## 25. 错误码设计

| error_code       | HTTP 状态码 | 说明                     |
| ---------------- | ----------- | ------------------------ |
| UNAUTHORIZED     | 401         | 未登录或 Access Key 无效 |

## 32. 关键注意事项

1. 这个系统的客户端都是内部后端服务，不是浏览器前端、小程序或外部用户客户端。
2. Runtime API 会返回真实上游 API Key，因此必须部署在内网。
3. 上游 API Key 需要可逆加密，Access Key 只需要 hash。
4. 业务服务应该使用 alias，不应该依赖真实模型名。
5. 配置中心只负责配置下发，不负责代理模型请求。
6. 第一版先把配置管理和安全鉴权做好，后续再考虑代理、fallback、限流、计费统计等高级能力。
